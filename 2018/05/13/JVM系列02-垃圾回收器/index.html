JVM系列02 - 垃圾回收器

  • Java
  • 2018-05-13

上一篇文章讲解了内存如何分配问题，这一篇讲一下内存如何回收。

什么样的对象是可以被回收的？

引用计数算法

每一个对象都有一个对应的引用计数器，当有地方引用时候，计数器加一；而当有地方的引用失效时，引用减一，当引用计数器为零时，认为当前对象可以被回收。

优点：实现起来简单高效

缺陷：无法解决循环引用的问题，可能导致内存泄漏

可达性分析算法

可达性分析算法基本的思路就是通过一系列的称为”GC Roots“的根对象作为起始节点，通过这些节点，根据引用关系向下搜索，如果搜索过程中没有某个对象不和任何的GC Roots相关联，可以认为该对象是不可达的，则证明此对象是不可能被使用的。JVM采用的正是这种可达性分析算法。

引用的分类

背景：如果我们希望描述一类对象：如果内存空间还足够的时候，就将这些对象保存中，如果内存空间很紧张的时候，就回收这些对象。

  • 强引用：只要强引用还在，对象永远不会回收。
  • 软引用：描述一些还有用，但是非必须的对象，在系统发生OOM前，触发这部分数据的回收。
  • 弱引用：描述非必须对象，比软引用更弱一些，被弱引用关联的对象只能生存到下一次垃圾收集发生为止。
  • 虚引用：最弱的一种引用，对内存回收没有影响，只是回收时GC会收到一个系统通知。

什么时候触发垃圾回收？

  • 程序调用System.gc()可以建议JVM进行垃圾回收，但是并不代表JVM立马就会进行垃圾回收的操作
  • JVM自身决定，原则是当内存不足时触发垃圾回收。首先说一下，gc什么时候触发Minor GC，对象分配优先在Eden进行，当Eden内存空间不足的时候，触发。其次，当老年带空间不足时，触发Full GC。

对象逃逸

GC在进行垃圾回收的时候，会调用finalize（）方法，这个时候如果给对象指定对应的引用，对象就会逃逸，但是限制只能逃逸一次。

对象如何回收？

可以认为，堆内存分代策略是数据存储的一种策略（数据结构），而下面介绍的三种算法时针对不同的分代区域采用的回收算法（算法），其目的是平衡时间复杂度和空间复杂度，从而提升程序性能。

分代收集

分代收集的背景：

  • 大部分的对象都是朝生晚死
  • 熬过垃圾收集次数越多的对象越难被收集

在这种背景之下，采用分代策略，对不同的区域有针对性的采取不同的回收算法。通常，分为Minor GC（新生代回收）、Major GC（老年代GC）、Full GC（整个Heap的GC）

清除算法

  • 标记-清除算法：分为标记数据的过程，以及清除数据的过程，会造成空间碎片。
  • 标记-复制算法：将内存分为两块相同大小的区域，一块存储对象，一块为空，回收时将无法回收的对象直接复制到空的区域，并顺序排列，空间复杂度较高，当然实际操作时候两块区域不会是1：1这样，如HotSpot的新生代就是这种算法，但是默认比例是8：1：1.
  • 标记-整理算法：标记清除算法的基础之上，最后还有一个空间整理的过程，时间复杂度较高。

常见的垃圾回收机以及原理

新生代收集器

  • Serial，单线程的收集器，回收收集的时候会触发”Stop the world“
  • ParNew，Serial收集器的多线程版本，除了JDK9+以外，只能与CMS配合使用，可以让垃圾回收和用户线程基本上同时运行。
  • Parallel Scavenge，基于标记-复制算法实现的收集器，能够并行的执行收集操作，其目的主要是为了提高吞吐量。

老年代收集器

  • CMS：获取最短时间收集为目标的收集器，基于标记-清除算法
  • Serial Old（MSC），单线程的收集器，使用标记-整理算法
  • Parallel Old，Parallel Scavenge的老年版本，多线程并发，基于标记整理算法实现

其他

  • G1，一个全面的垃圾收集器，不需要和其他垃圾收集器进行配合。主要面向服务端应用的垃圾收集器，java 9 成为服务端模式下的默认垃圾回收器。
