JVM系列01 - 内存管理

  • Java
  • 2018-05-09

Java语言这么流行最主要的原因之一就是：自动分配内存，自动回收内存，即JVM的内存管理机制。而C、C++类的语言内存的分配和回收都是由程序员控制的，很容易出现内存溢出、内存泄漏等问题。内存自动分配和回收机制方便管理的同时，也带来了新的问题，如果出现内存泄漏问题，排查起来就会异常艰辛，所以我们需要对JVM相关技术有一定的了解。

为什么需要学习JVM？

  • 系统出现OOM，频繁Full GC的时候不会束手无策，可以有针对性的进行排查和处理。
  • 通过JVM调优，可以优化程序性能。
  • Java中的类库，很多事基于JVM优化实现的，如一些语法糖，类加载器等。
  • 面试官喜欢问，嘿嘿，你要是一直不跳槽的话，当我没说。

运行时数据区域划分

JVM在启动到销毁的过程中，会将运行中的程序的内存划分为不同的区域，并且每个区域的生命周期也不一下，如下图所示。

程序计数器

程序计数器是线程私有的，每一个线程的工作内存中都会有一个程序计数器，是JVM中为一个没有定义OOM的区域。可以理解为程序的行号记录器，记录程序执行的行数，方便程序的跳转等指令的执行。

虚拟机栈、本地方法栈

线程私有的区域，生命周期和线程相同。每个方法执行时，都会创建一个栈帧，用于存储局部变量、操作数栈、动态连接、方法出口等信息。

每一个方法被调用到执行完毕的过程，就对应着这个方法的栈帧在虚拟机栈或者本地方法栈中出栈到入栈的过程。

如果线程请求的栈深度大于虚拟机所允许的深度，将会抛出StackOverflowError；如果Java虚拟机栈容量可以动态扩展，当栈扩展到无法申请到足够的内存的时候就会抛出OutOfMemoryError异常。

Java堆（Heap）

线程共享的区域，生命周期和JVM进程相同。几乎所有的对象实例和数组都在堆上分配，堆内存是GC回收的最主要的区域。从内存回收的角度来看，现代垃圾收集器大部分都是基于分代收集器理论设计的，所以堆内存经常出现”新生代”、”老年带”、”永久代”、”Eden”、”From Survivor”、”To Survivor”等概念。

堆内存在物理上可以设计为使用不连续的内存空间，但是在逻辑上应该是连续的，这点就像我们用磁盘去存储文件一样，并不考虑每个文件都连续存放。但对于大对象（典型的如数组），多数虚拟机实现处于简单、存储高效的考虑，很可能会要求连续的存储空间。

堆可以设置为大小固定的，也可以设置为大小可扩展的，-Xms，-Xmx，-Xmn，如果在堆内存上无法完成实例的分配，并且堆无法再扩展的时候，JVM将会抛出OutOfMemoryError的异常。

方法区（Method Area）

方法区是线程共享的区域，和JVM进程的生命周期相同，用于存储已被虚拟机加载的类型信息、常量、静态变量、即时编译器编译后的代码缓存等数据。当常量池无法申请到内存时，会抛出OutOfMemoryError异常。

直接内存（Direct Memory）

直接内存并不是运行时数据区的一部分，也不是JVM规范中定义的内存区域，但是这部分内存也被频繁的使用，而且也可能导致OutOfMemoryError异常出现。

NIO可以直接调用Native函数进行堆外内的分配，然后通过一个存储在堆内存的DirectByteBuffer对象作为这块内存的引用进行操作，而不需要来回的在堆和native堆进行数据的复制。

直接内存不受堆内存大小的限制，但是肯定会受物理内存的限制，所以一般设置堆内存的大小时候，需要预留一部分内存，而不是设置和物理内存一样大小。

对象的创建流程

类加载

当JVM遇到一条new字节码的时候，首先会去检查这个指令对应的参数能否在常量池中定位到一个类的符号引用，并且检查这个符号引用代表的类是否已被加载、解析和初始化过。如果没有，那么首先触发类加载的过程。

分配内存

在类检查完毕后，接下里JVM会为对象分配内存，对象所需的内存大小在类加载完毕后便是可以确认的，为对象分配空间的任务便等同于把一块确定大小的内存从堆中划分出来。

如何从堆中划分内存给对象通常由GC算法决定，使用标记-复制、标记-整理算法的GC的堆通常都是连续的，所以使用简单的”指针碰撞“即可，而使用标记-清除算法的GC的堆通常是不连续的，通常使用”空闲列表“的方式。

如何保证分配内存操作是线程安全的？

  • 采用同步的方式，JVM采用的是CAS + 失败重试保证线程安全。
  • 内存的分配动作按照线程划分到不同的空间中进行，即每一个线程在Java堆中预先分配一小块内存，即本地线程分配缓冲区（Thread Local Allocation Buffer），哪一个线程要分配内存，就在哪一个线程的本地缓冲区中分配，只有本地缓存区使用完了，分配新的缓冲区时才需要同步锁定。

对象头初始化

分配完内存后，JVM需要对对象头进行初始化，如对象的类型元数据信息、对象的hashCode（对象的哈希码会延迟到真正调用Object::hashCode（）方法时才计算）、对象的GC分代年龄等信息。

对象的内存布局

在HotSpot虚拟机中，对象在堆内存中可以划分为三个部分: 对象头(Header)、实例数据(Instance Data)和对象对其(Padding)。

对象头主要包括两部分的信息:

  • 对象自身运行时数据（Mark Word），如hashCode、GC分代年龄、锁状态标记、线程持有的锁、偏向线程ID、偏向时间戳等。
  • 类型指针，即指向它的类型元数据的指针。

对象的访问定位

创建对象自然是为了后续使用该对象，通常使用栈上的reference数据来操作具体的对象，这里的reference分为句柄和指针，通常的HotSpot使用的是指针的方式。
