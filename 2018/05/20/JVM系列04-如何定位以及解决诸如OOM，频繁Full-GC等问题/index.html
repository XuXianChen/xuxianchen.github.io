JVM系列04 - JVM性能调优实战篇

  • Java
  • 2018-05-20

日常的开发过程中，无需可以关注JVM，但是如果生产环境出现JVM相关故障，如果对JVM的原理和相关调优一无所知，那么久很难进行排查和解决问题。

JVM内存回收的流程

新生代为啥会有两个survivor？

背景

  • 新生代通常是采用标记-复制算法进行垃圾回收
  • 新生代的对象通常都是朝生晚死的

新生代将内存分为Eden、from、to三个区域，通常默认的大小为8：1：1，新对象在Eden和from区域分配，当发生minor gc的时候，通过复制算法将Eden和from的没有回收的对象复制到to区域，为了保证对象可以继续在Eden区域分配对象，将现有的from区和to区域进行互换，那么现在的to区域又是一块完整的空闲区域，可以完成后续的复制算法。

JVM常出现的异常有哪些？

频繁的GC

Minor GC

通常频繁的Minor GC是由于新生代的内存分配过小，发生这种情况需要将新生代的空间分配大一些，但是新生代分配过大会造成Minor GC回收的时间变长，所以遇到这种问题需要特别小心，即时进行了优化，也需要进行大量的测试来进行验证。

Full GC（通常一天超过一次以上的Full GC就算是比较频繁了）

  • 创建了大量的大对象，超过了Minor GC的阈值，直接进入了老年代，编程过程中注意一下，可以将大对象拆分成几个小对象来实现。
  • 通过heap dump文件，判断内存中是否存在大量相同的对象，并且每次Full GC回收的对象不多。

常见解决方案：

  • 减少大对象创建
  • 适当增加堆内存大小
  • 选择合适的垃圾收集器（如CMS，G1等）

OOM（内存泄漏、内存溢出）、死锁、死循环等问题造成CPU负载过高

通常都是程序BUG，重点是如何排除。

  • 使用top命令查看系统各项指标。
  • top -Hp pid 查看具体进程的线程使用情况。
  • 通过上面的命令，如果分析出是Java线程原因，可以使用jstack jpid | grep tid 的方式将占有CPU最高的线程数据进行输出。
  • 然后分析线程堆栈基本上可以定位到问题代码的所在位置
  • heap dump日志，通过相关可视化工具MAT，visualVM进行离线分析，找出内存占用较高的对象，并找到什么地方进行了创建和引用。
  • 配合GC日志，GC日志生成可以通过JVM参数进行配置，通过相关可视化工具GC Easy进行分析
